## 2. Pre-existing RTL issue (fixed before testbench run)

### Symptom

`iverilog -g2012` rejected all six sub-module files with 192 errors of the form:

```
reed_solomon_ecc_w8.v:82: error: All but the final index in a chain of indices
must be a single value, not a range.
reed_solomon_ecc_w8.v:82: error: Unable to elaborate r-value: codeword_in['sd7:'sd0]['sd0]
```

### Repro

```bash
iverilog -g2012 -o /dev/null \
    results/issue-11/rtl/reed_solomon_ecc_w8.v
```

### Root cause

The generator that produced the attached RTL emitted `codeword_in[7:0][N]` for the
syndrome-shift stage instead of the plain bit-select `codeword_in[N]`.
Chaining a part-select (`[7:0]`) then a bit-select (`[N]`) is illegal in Verilog/SV.

### Fix applied

A single `sed` substitution on all six `_w*.v` files:

```
s/codeword_in\[7:0\]\[([0-9]*)\]/codeword_in[\1]/g
```

This converts every `codeword_in[7:0][N]` → `codeword_in[N]`, which is semantically
equivalent (S0 of the syndrome chain just copies the first byte of the codeword verbatim).
No functional logic was altered.

Evidence: `results/issue-11/build/compile.log` (zero errors/warnings after fix).

---

## 3. Root cause analysis

The attached zip was generated by an automated RTL generator that incorrectly emitted
chained range+bit selects in the syndrome calculation loop. The issue is purely
syntactic—the intended behaviour (copy `codeword_in[N]` into the first shift-register
stage) is identical to the corrected form.

## 6. Coverage description

The testbench exercises all six `DATA_WIDTH` configurations of the `reed_solomon_ecc`
wrapper, covering the following behaviours:

| Area | What is tested |
|------|----------------|
| **Reset** | Active-low asynchronous reset clears all registered outputs across all DUTs |
| **Encode (nominal)** | Round-trip for representative patterns: 0x00, 0xFF, 0xAB, 0xA (nibble), 0xBEEF, 0xDEADBEEF, 0xCAFEBABEDEAD1234, 128-bit pattern |
| **Systematic property** | Data bytes appear at the LSB of the codeword after encode |
| **Decode (clean)** | Feeding a clean encoded codeword back returns original data with `error_detected=0` |
| **Error detection** | Flipping a parity byte or a data byte causes `error_detected=1` for DATA_WIDTH 4, 8, 16, 32, 64, 128 |
| **`error_corrected` placeholder** | Confirmed always 0 even on corrupted codewords (RTL documents this as a placeholder) |
| **valid_out timing** | Asserts exactly one cycle after enable; deasserts on the next idle cycle |
| **Back-to-back encode** | Three consecutive encode operations with different data values |
| **Simultaneous enable** | Both `encode_en` and `decode_en` high in same cycle; `valid_out` still fires |
| **Async reset mid-op** | Reset pulsed during a live encode; outputs clear immediately; normal operation resumes |

## 7. Gaps / limitations

| Gap | Notes |
|-----|-------|
| **Error correction not implemented** | The RTL itself marks `error_corrected` as a placeholder (always 0); actual symbol-level correction is not testable until the RTL implements it |
| **Single-symbol vs two-symbol error patterns** | Only single-byte-flip errors are tested; RS(T=2) can potentially correct 1 symbol and detect 2, but the decoder does not yet correct so boundary cases are not exercised |
| **Randomised / exhaustive stimulus** | No constrained-random or exhaustive covering of all 256 GF symbol values per config |
| **Upper bits of 160-bit bus** | Tests verify the meaningful lower slice; the upper zero-padding bits are not checked explicitly on every operation |
| **Power-on (pre-reset) behaviour** | Not checked — simulation starts with `rst_n=0`, which is the intended safe power-on state |
